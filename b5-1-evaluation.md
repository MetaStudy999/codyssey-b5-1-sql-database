### 1-2. `FK`를 사용한 `1:N` 관계가 최소 2개 이상 존재하고, 없는 값 참조가 실제로 막히는가?

- [x] 완료

**요약:**  
`FK`를 사용해 3개의 `1:N` 관계를 만들었고, SQLite의 `PRAGMA foreign_keys = ON` 설정으로 없는 값을 참조하는 입력이 막히도록 했다.

**자세한 설명:**  
이 DB에는 `category 1:N book`, `member 1:N rental`, `book 1:N rental` 관계가 있다. 예를 들어 한 명의 회원은 여러 번 책을 빌릴 수 있으므로 `member`가 1이고 `rental`이 N이다. `rental.member_id`는 반드시 `member.member_id`에 있는 값만 저장할 수 있다. 만약 존재하지 않는 `member_id = 999`로 대여 기록을 넣으려고 하면 `FOREIGN KEY constraint failed`가 발생한다. 즉, 잘못된 대여 기록이 DB에 들어가는 것을 `FK`가 막아 준다.

**핵심 키워드:** `FK`, `FOREIGN KEY`, `1:N`, `참조 무결성`, `PRAGMA foreign_keys`

## 항목 3. SQL 개념 이해

### 3-1. 데이터베이스가 엑셀과 무엇이 다른지, 왜 테이블을 나눠서 저장하는지 비교하며 설명할 수 있는가?

- [x] 설명 가능

**요약:**  
엑셀은 한 표에 데이터를 직접 정리하는 데 편하고, 데이터베이스는 여러 테이블의 관계와 규칙을 안전하게 관리하는 데 강하다.

**자세한 설명:**  
엑셀에서도 표를 만들 수 있지만, 다른 표의 값만 참조하도록 강제하거나 잘못된 관계를 자동으로 막는 기능은 제한적이다. 반면 데이터베이스는 `PK`, `FK`, `NOT NULL`, `UNIQUE`, `CHECK` 같은 규칙을 사용해 잘못된 데이터가 들어오는 것을 막을 수 있다. 테이블을 나누는 이유는 중복을 줄이고 변경을 쉽게 하기 위해서다. 예를 들어 회원 이메일이 바뀌면 `member` 테이블 한 곳만 수정하면 되고, 기존 대여 기록은 `member_id`로 계속 연결된다.

**핵심 키워드:** `관계형 데이터베이스`, `엑셀`, `무결성`, `제약조건`, `중복 제거`

### 3-2. `PK`와 `FK`의 역할을 구분하고, `1:N` 관계가 데이터를 어떻게 연결하는지 본인 스키마 기준으로 보여줄 수 있는가?

- [x] 설명 가능

**요약:**  
`PK`는 자기 테이블의 행을 구분하는 값이고, `FK`는 다른 테이블의 `PK`를 참조해 테이블 사이를 연결하는 값이다.

**자세한 설명:**  
`member.member_id`는 회원을 구분하는 `PK`이다. `rental.member_id`는 이 값을 참조하는 `FK`이다. 그래서 `rental`에는 회원 이름을 직접 반복 저장하지 않고 `member_id`만 저장한다. 회원 이름이 필요할 때는 `JOIN`으로 `member` 테이블과 연결한다. 이 구조 덕분에 한 명의 회원이 여러 대여 기록을 가져도 회원 정보는 한 번만 저장되고, 대여 기록만 여러 행으로 쌓인다.

**핵심 키워드:** `PK`, `FK`, `JOIN`, `1:N`, `관계 연결`, `참조`

---

### 3-3. `INNER JOIN`과 `LEFT JOIN`의 차이를 실행 결과를 보며 짚어줄 수 있는가?

- [x] 설명 가능

**요약:**  
`INNER JOIN`은 양쪽 테이블에 모두 연결되는 데이터만 보여주고, `LEFT JOIN`은 왼쪽 테이블의 데이터를 모두 남긴다.

**자세한 설명:**  
Q05, Q06, Q07에서는 `INNER JOIN`을 사용했다. 예를 들어 Q05는 `rental`, `member`, `book`이 모두 연결되는 대여 기록만 보여준다. 연결되는 회원이나 책이 없다면 결과에 나오지 않는다. 반면 Q08은 `member LEFT JOIN rental`을 사용한다. 이 경우 대여 기록이 없는 회원도 결과에 포함되고, 대여 기록이 없으면 `COUNT(r.rental_id)`가 0으로 나온다. 즉, 누락된 관계까지 확인하고 싶을 때 `LEFT JOIN`이 유용하다.

**핵심 키워드:** `INNER JOIN`, `LEFT JOIN`, `매칭`, `NULL`, `대여 없는 회원`

---

### 3-4. `GROUP BY`와 집계 함수(`COUNT`, `SUM`, `AVG`)가 어떻게 동작하는지 쿼리 결과를 보며 이야기할 수 있는가?

- [x] 설명 가능

**요약:**  
`GROUP BY`는 같은 기준의 행을 묶고, 집계 함수는 묶인 데이터의 개수, 합계, 평균을 계산한다.

**자세한 설명:**  
Q09는 회원별로 대여 기록을 묶고 `COUNT`로 대여 횟수, `SUM`으로 수수료 합계를 계산한다. Q10은 카테고리별로 도서를 묶고 `COUNT`로 도서 수, `AVG`로 평균 가격을 계산한다. Q11은 대여 상태별로 묶어 상태별 건수와 수수료 합계를 계산한다. 입문자 관점에서는 `GROUP BY`를 "같은 이름표를 가진 행끼리 한 바구니에 담는 과정"으로 이해하면 쉽다. 그 바구니 안에서 `COUNT`, `SUM`, `AVG`가 각각 개수, 합계, 평균을 구한다.

**핵심 키워드:** `GROUP BY`, `COUNT`, `SUM`, `AVG`, `집계`, `그룹`

## 항목 4. 문제 해결 과정

### 4-1. 작성한 쿼리 중 가장 복잡했던 쿼리를 선택하고, 어떻게 풀었는지 단계별로 설명할 수 있는가?

- [x] 설명 가능

**요약:**  
가장 복잡했던 쿼리는 Q07이며, 연체된 대여 기록을 회원, 도서, 카테고리 정보와 함께 보여주는 4개 테이블 조인 쿼리이다.

**자세한 설명:**  
Q07은 `rental`, `member`, `book`, `category`를 함께 사용한다. 먼저 기준 테이블을 `rental`로 잡았다. 이유는 연체 여부인 `status = 'OVERDUE'`가 대여 기록에 있기 때문이다. 다음으로 `rental.member_id = member.member_id` 조건으로 회원 이름을 가져왔다. 그다음 `rental.book_id = book.book_id` 조건으로 도서 제목을 가져왔다. 마지막으로 `book.category_id = category.category_id` 조건으로 카테고리 이름을 가져왔다. 그리고 `WHERE r.status = 'OVERDUE'`로 연체 기록만 필터링하고, `ORDER BY r.due_date ASC`로 오래된 반납기한부터 정렬했다.

**단계별 풀이:**

1. 연체 상태가 저장된 `rental`을 출발점으로 정한다.
2. `member`를 조인해 회원 이름을 붙인다.
3. `book`을 조인해 도서 제목을 붙인다.
4. `category`를 조인해 도서 분류를 붙인다.
5. `WHERE`로 `OVERDUE` 상태만 남긴다.
6. `ORDER BY`로 반납기한이 빠른 순서대로 정렬한다.

**핵심 키워드:** `다중 JOIN`, `INNER JOIN`, `WHERE`, `ORDER BY`, `OVERDUE`, `별칭(alias)`

### 4-2. 미션 수행 중 가장 어려웠던 부분과 해결 방법을 구체적으로 말할 수 있는가?

**요약:**  
가장 어려웠던 부분은 테이블을 어떤 기준으로 나눌지와 `FK` 입력 순서를 맞추는 것이었다.

**자세한 설명:**  
처음에는 회원명, 책 제목, 대여일을 한 테이블에 모두 넣으면 쉬워 보인다. 하지만 그렇게 하면 같은 회원과 책 정보가 반복되고, 나중에 수정할 때 여러 곳을 고쳐야 한다. 그래서 "변하지 않는 정보"와 "발생하는 사건"을 나누는 방식으로 해결했다. 회원과 책은 각각 독립된 정보이고, 대여는 회원과 책이 만나는 사건이므로 `rental` 테이블로 분리했다. 또 `FK`가 있는 테이블은 부모 데이터가 먼저 있어야 하므로 `member`, `category`를 먼저 넣고, 그다음 `book`, 마지막으로 `rental`을 입력했다. 데이터 변경 쿼리인 `UPDATE`, `DELETE`는 실수로 많은 행을 바꾸지 않도록 `WHERE rental_id = ...`처럼 `PK` 기준 조건을 사용했다.

보너스 과제로 JOIN과 서브쿼리 비교, FK 오류 데모, 미니 리포트 3개를 작성했다.

**자세한 설명:**  
첫 번째 보너스에서는 "AI 카테고리 도서를 대여한 회원 찾기"를 `JOIN` 방식과 서브쿼리 방식으로 각각 풀었다. `JOIN` 방식은 테이블 관계를 한눈에 보기 좋고, 서브쿼리 방식은 안쪽 쿼리부터 조건을 좁혀 가는 흐름을 이해하기 좋다. 두 번째 보너스에서는 존재하지 않는 `member_id = 999`로 대여 기록을 넣으려 하면 `FK` 오류가 발생한다는 점을 기록했다. 세 번째 보너스에서는 월별 대여 건수, 인기 도서 TOP 5, 연체율이 높은 회원을 구하는 미니 리포트를 만들었다. 이 지표들은 실제 도서관 운영에서 대여 추이, 인기 도서 구매, 연체 관리에 활용할 수 있다.
