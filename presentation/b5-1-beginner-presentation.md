## Slide 5. 입문자 개념: 엑셀과 데이터베이스의 차이

### 슬라이드 목표

입문자가 왜 데이터베이스를 써야 하는지 이해하도록 합니다.

### 입문자용 쉬운 설명

엑셀은 사람이 눈으로 보고 직접 관리하기 좋은 도구입니다.  
하지만 데이터가 많아지고, 서로 연결된 정보가 많아지면 실수가 생깁니다.  
데이터베이스는 이런 실수를 줄이기 위해 **관계**와 **규칙**을 사용합니다.

### 발표 멘트

엑셀과 데이터베이스의 차이는 단순히 데이터 양이 아닙니다.  
핵심 차이는 **관계를 표현하고 규칙을 강제할 수 있는가**입니다.  
엑셀에 회원명과 책 제목을 한 표에 계속 적으면 중복이 많아집니다.  
반면 데이터베이스는 회원은 `member`, 책은 `book`, 대여는 `rental`로 나누고, FK로 연결할 수 있습니다.

### 핵심 키워드

- 엑셀
- 관계형 데이터베이스
- 중복 제거
- 제약조건
- 무결성
- 관계

### 실습/시현

직접 보여줄 필요는 없지만, 평가 질문 대비로 아래 예시를 말로 설명합니다.

```text

```

### 간단 요약

> 엑셀은 한 표 관리에 편하고, 데이터베이스는 관계와 규칙 관리에 강합니다.

## Slide 6. 전체 테이블 구조

### 슬라이드 목표

회원 서랍, 책 서랍, 카테고리 서랍, 대여 기록 서랍을 따로 만든 것입니다.

특히 `rental`은 회원과 책이 만나는 지점이므로 가장 중요한 사건 테이블입니다.

## Slide 7. ERD 관계 설명

### 슬라이드 목표

테이블 간 관계를 시각적으로 설명합니다.

### 입문자용 쉬운 설명

ERD는 데이터베이스의 지도입니다.  
어떤 테이블이 어떤 테이블과 연결되는지 그림으로 보여줍니다.

### 발표 멘트

이 프로젝트의 관계는 세 가지입니다.  
하나의 카테고리는 여러 책을 가질 수 있으므로 `category 1:N book`입니다.  
한 명의 회원은 여러 대여 기록을 가질 수 있으므로 `member 1:N rental`입니다.  
한 권의 책도 여러 번 대여될 수 있으므로 `book 1:N rental`입니다.

### 관계 정리

| 관계 | 의미 |
|---|---|
| `category 1:N book` | 하나의 카테고리에 여러 책이 속한다 |
| `member 1:N rental` | 한 명의 회원이 여러 번 책을 빌릴 수 있다 |
| `book 1:N rental` | 한 권의 책이 여러 번 대여될 수 있다 |

### 핵심 키워드

- ERD
- 1:N 관계
- 부모 테이블
- 자식 테이블
- 참조
- 관계 모델링

### 실습/시현

```bash
cat docs/ERD.md
```

확인할 내용:

- `MEMBER ||--o{ RENTAL`
- `BOOK ||--o{ RENTAL`
- `CATEGORY ||--o{ BOOK`

### 간단 요약

> 이 프로젝트에는 3개의 1:N 관계가 있으며, 요구사항의 2개 이상 조건을 충족합니다.

### 평가 연결

- 항목 1-2: FK를 사용한 1:N 관계 2개 이상
- 항목 2-2: 1:N 관계의 도메인 의미 설명

## Slide 8. 테이블을 나눈 이유

### 슬라이드 목표

정규화와 역할 분리의 필요성을 입문자도 이해하게 합니다.

### 입문자용 쉬운 설명

한 노트에 모든 정보를 반복해서 쓰면 나중에 고치기 어렵습니다.  
회원 정보는 회원 노트에, 책 정보는 책 노트에, 빌린 기록은 대여 노트에 쓰는 것이 더 안전합니다.

### 발표 멘트

처음에는 회원명, 이메일, 책 제목, 카테고리명, 대여일을 한 테이블에 모두 넣으면 쉬워 보입니다.  
하지만 그렇게 하면 같은 회원 정보와 책 정보가 계속 반복됩니다.  
그래서 변하지 않는 정보인 회원과 책은 따로 저장하고, 대여라는 사건은 `rental`에 저장했습니다.  
이 구조 덕분에 회원 이메일이 바뀌어도 `member` 테이블 한 곳만 수정하면 됩니다.

### 핵심 키워드

- 정규화
- 중복 제거
- 역할 분리
- 사건 테이블
- 수정 용이성
- 데이터 일관성

### 실습/시현

말로 비교합니다.

```text
나쁜 구조:
대여 테이블 안에 회원 이름, 이메일, 책 제목, 카테고리를 모두 반복 저장

좋은 구조:
member: 회원 정보
book: 책 정보
```

PK는 각 데이터의 고유 번호입니다.  
FK는 다른 테이블의 고유 번호를 가져와서 “이 데이터와 연결되어 있다”라고 표시하는 값입니다.

FK는 다른 테이블의 PK를 참조하는 값입니다.  
예를 들어 `rental.member_id`는 `member.member_id`를 참조해서, 이 대여 기록이 어떤 회원의 기록인지 알려줍니다.

### 핵심 키워드

- FK

- 참조
- 연결

```sql
member_id INTEGER PRIMARY KEY

```

### 간단 요약

> PK는 행을 구분하고, FK는 테이블 사이를 연결합니다.

## Slide 10. 제약조건 설명

### 슬라이드 목표

잘못된 데이터 입력을 막는 장치를 설명합니다.

### 입문자용 쉬운 설명

제약조건은 데이터베이스의 안전벨트입니다.  
아무 값이나 들어오지 못하게 막아줍니다.

### 발표 멘트

데이터베이스는 데이터를 저장하는 것뿐 아니라, 잘못된 데이터가 들어오지 못하게 막아야 합니다.  
그래서 `NOT NULL`, `UNIQUE`, `CHECK`, `DEFAULT`, `FOREIGN KEY`를 사용했습니다.  
예를 들어 이메일은 중복되면 안 되므로 `UNIQUE`, 대여 상태는 정해진 값만 가능해야 하므로 `CHECK`, 존재하지 않는 회원으로 대여 기록을 만들면 안 되므로 `FOREIGN KEY`를 사용했습니다.

### 제약조건 정리

| 제약조건 | 쉬운 설명 | 프로젝트 예시 |
|---|---|---|
| `NOT NULL` | 비워두면 안 됨 | 이름, 이메일, 도서명 |
| `UNIQUE` | 중복되면 안 됨 | 이메일, ISBN, 카테고리명 |
| `CHECK` | 정해진 범위만 허용 | 상태값, 가격, 재고 |
| `DEFAULT` | 기본값 자동 입력 | 상태 기본값 |
| `FOREIGN KEY` | 없는 부모 참조 금지 | 없는 회원 대여 방지 |

### 핵심 키워드

- 제약조건
- NOT NULL
- UNIQUE
- CHECK
- DEFAULT
- FOREIGN KEY
- 데이터 무결성

## Slide 18. 대표 쿼리 2: Q08 LEFT JOIN

### 슬라이드 목표

INNER JOIN과 LEFT JOIN 차이를 실전 예제로 설명합니다.

### 입문자용 쉬운 설명

INNER JOIN은 양쪽에 모두 있는 데이터만 보여줍니다.  
LEFT JOIN은 왼쪽 테이블의 데이터는 일단 모두 보여줍니다.  
그래서 대여 기록이 없는 회원도 볼 수 있습니다.

### 발표 멘트

Q08은 `member LEFT JOIN rental`을 사용합니다.  
이 쿼리의 목적은 대여 기록이 없는 회원까지 포함해서 회원별 대여 횟수를 확인하는 것입니다.  
INNER JOIN을 사용하면 대여 기록이 없는 회원은 결과에서 사라집니다.  
하지만 LEFT JOIN을 사용하면 그런 회원도 남고, `COUNT(r.rental_id)`가 0으로 나옵니다.

### 핵심 키워드

- LEFT JOIN
- INNER JOIN
- NULL
- COUNT
- 대여 없는 회원
- 누락 데이터 확인

### 실습/시현

```sql
SELECT m.member_id, m.name, COUNT(r.rental_id) AS rental_count
FROM member m
LEFT JOIN rental r ON m.member_id = r.member_id
GROUP BY m.member_id, m.name
ORDER BY rental_count ASC, m.member_id ASC;
```

### 간단 요약

> LEFT JOIN은 연결된 기록이 없는 데이터까지 확인할 때 유용합니다.

### 평가 연결

- 항목 1-4: LEFT JOIN 1개 이상 포함
- 항목 3-3: INNER JOIN과 LEFT JOIN 차이

# 7분 발표 추천 대본

안녕하세요. B5-1 “SQL로 만드는 나만의 데이터베이스” 미션 발표를 시작하겠습니다.  
제 프로젝트 주제는 도서 대여 관리 데이터베이스입니다.

이 미션은 단순히 SQL 문법을 작성하는 것이 아니라, 도메인 데이터를 테이블로 나누고, PK와 FK 관계를 만들고, 실제 요구사항을 SQL 쿼리로 해결하는 과제입니다.

저는 도서관에서 회원이 책을 빌리고 반납하는 상황을 모델링했습니다.  
테이블은 총 4개입니다.  
`member`는 회원 정보, `category`는 도서 분류, `book`은 도서 정보, `rental`은 대여 기록입니다.

이렇게 나눈 이유는 중복을 줄이고 관계를 명확하게 표현하기 위해서입니다.  
엑셀처럼 한 표에 회원명, 이메일, 책 제목, 대여일을 모두 넣으면 같은 정보가 반복됩니다.  
그래서 회원 정보와 도서 정보는 따로 저장하고, 대여라는 사건은 `rental` 테이블에 저장했습니다.

이 프로젝트에는 3개의 1:N 관계가 있습니다.  
하나의 카테고리는 여러 책을 가질 수 있으므로 `category 1:N book`입니다.  
한 명의 회원은 여러 대여 기록을 가질 수 있으므로 `member 1:N rental`입니다.  
한 권의 책도 여러 번 대여될 수 있으므로 `book 1:N rental`입니다.

스키마에서는 모든 테이블에 PK를 두었고, `book.category_id`, `rental.member_id`, `rental.book_id`를 FK로 설정했습니다.  
또 이메일과 ISBN에는 `UNIQUE`, 상태값과 가격에는 `CHECK`, 필수값에는 `NOT NULL`을 사용해 잘못된 데이터가 들어오지 않도록 했습니다.

샘플 데이터는 `member` 10행, `category` 10행, `book` 15행, `rental` 20행을 입력했습니다.  
FK 관계가 있기 때문에 부모 테이블인 `member`, `category`를 먼저 넣고, 그다음 `book`, 마지막으로 `rental`을 입력했습니다.

핵심 쿼리는 총 15개입니다.  
Q01부터 Q04는 기본 조회, Q05부터 Q08은 JOIN, Q09부터 Q11은 집계, Q12는 서브쿼리, Q13은 UPDATE, Q14는 DELETE, Q15는 인덱스입니다.

가장 복잡했던 쿼리는 Q07입니다.  
Q07은 연체된 대여 기록을 회원명, 도서명, 카테고리명과 함께 보여주는 4개 테이블 JOIN 쿼리입니다.  
연체 상태가 `rental`에 있으므로 `rental`을 기준으로 잡고, `member`, `book`, `category`를 차례로 조인했습니다.

인덱스는 `rental(member_id, due_date)`에 만들었습니다.  
도서관에서는 특정 회원의 대여 기록을 반납기한 순으로 확인하는 일이 자주 있을 수 있기 때문입니다.

실행은 `scripts/run_all.sh`로 자동화했습니다.  
이 스크립트는 DB 생성, 시드 데이터 입력, 검증 SQL 실행, 핵심 쿼리 실행, 결과 파일 저장까지 한 번에 처리합니다.  
또 UPDATE와 DELETE는 임시 DB 복사본에서 실행되도록 해서 원본 DB가 훼손되지 않게 했습니다.

보너스로는 JOIN과 서브쿼리 비교, FK 오류 데모, 미니 리포트 3개를 작성했습니다.  
특히 존재하지 않는 `member_id = 999`를 참조하면 `FOREIGN KEY constraint failed`가 발생하도록 확인했습니다.

이번 프로젝트를 통해 데이터베이스의 핵심은 단순히 데이터를 많이 저장하는 것이 아니라, 데이터를 관계와 규칙으로 안전하게 관리하는 것이라는 점을 배웠습니다.  
이상으로 발표를 마치겠습니다.

# 발표자가 기억해야 할 핵심 문장 5개

1. 데이터베이스의 핵심은 데이터 양이 아니라 **관계와 규칙**입니다.

4. JOIN은 나누어진 테이블을 다시 의미 있게 연결하는 방법입니다.
5. 이 프로젝트는 SQL 문법 연습을 넘어, 서비스 데이터를 구조화하는 첫 훈련입니다.
