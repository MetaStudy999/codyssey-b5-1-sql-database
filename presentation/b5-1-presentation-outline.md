## 5. 입문자 개념: 엑셀과 DB 차이

입문자에게 가장 먼저 필요한 개념 슬라이드입니다.

설명 포인트:

| 구분 | 엑셀 | 관계형 DB |
|---|---|---|
| 저장 방식 | 한 표 중심 | 여러 테이블 분리 |
| 관계 표현 | 약함 | PK/FK로 명확히 표현 |
| 오류 방지 | 수동 확인 중심 | 제약조건으로 자동 방지 |
| 조회 방식 | 필터·정렬 중심 | SQL로 검색·조인·집계 |

핵심 문장:

> 엑셀은 한 표에 데이터를 정리하기 쉽고, 데이터베이스는 여러 테이블의 관계와 규칙을 안전하게 관리하는 데 강합니다.

## 6. 전체 테이블 구조

4개 테이블의 역할을 한눈에 보여줍니다.

| 테이블 | 역할 |
|---|---|
| `member` | 도서관 회원 정보 |
| `category` | 도서 분류 정보 |
| `book` | 도서 정보 |
| `rental` | 회원이 책을 빌린 대여 기록 |

핵심 문장:

> `rental`은 회원과 도서를 연결하는 사건 테이블입니다.

---

## 7. ERD 관계 설명

ERD 또는 간단한 관계 그림을 넣습니다.

관계:

- `category 1:N book`
- `member 1:N rental`
- `book 1:N rental`

설명 문장:

> 하나의 카테고리에는 여러 권의 책이 들어갈 수 있고, 한 명의 회원은 여러 대여 기록을 가질 수 있으며, 한 권의 책도 여러 번 대여될 수 있습니다.

---

## 8. 테이블을 나눈 이유

발표에서 가장 중요한 모델링 설명입니다.

설명 포인트:

- 한 테이블에 모두 넣으면 회원명, 이메일, 도서명이 반복된다.
- 회원 정보가 바뀌면 여러 행을 수정해야 한다.
- 그래서 변하지 않는 정보와 발생하는 사건을 나누었다.
- `rental`은 “누가 어떤 책을 언제 빌렸는가”를 저장한다.

---

## 9. PK와 FK 쉽게 이해하기

입문자용 비유를 사용합니다.

| 개념 | 쉬운 설명 | 프로젝트 예시 |
|---|---|---|
| PK | 한 행을 구분하는 고유 번호 | `member.member_id` |
| FK | 다른 테이블과 연결하는 참조값 | `rental.member_id` |

핵심 문장:

> PK는 자기 테이블의 주민등록번호 같은 값이고, FK는 다른 테이블의 PK를 가리키는 연결선입니다.

---

## 10. 제약조건 설명

스키마 품질을 설명하는 슬라이드입니다.

포함할 제약조건:

- `NOT NULL`: 반드시 값이 있어야 함
- `UNIQUE`: 중복 금지
- `CHECK`: 허용된 값만 저장
- `DEFAULT`: 기본값 자동 입력
- `FOREIGN KEY`: 존재하는 부모 데이터만 참조

프로젝트 예시:

- 회원 이메일 중복 방지
- ISBN 중복 방지
- 대여 상태값 제한
- 음수 가격·음수 재고 방지
- 존재하지 않는 회원으로 대여 기록 생성 방지

## 17. 대표 쿼리 1: Q07 다중 JOIN

가장 중요한 심화 설명 슬라이드입니다.

설명 순서:

1. 기준 테이블을 `rental`로 잡는다.
2. `member`를 조인해 회원 이름을 가져온다.
3. `book`을 조인해 도서 제목을 가져온다.
4. `category`를 조인해 카테고리명을 가져온다.
5. `WHERE r.status = 'OVERDUE'`로 연체 기록만 남긴다.
6. `ORDER BY r.due_date ASC`로 반납기한이 빠른 순서로 정렬한다.

핵심 문장:

> Q07은 대여 기록이라는 사건을 중심으로 회원, 도서, 카테고리 정보를 한 번에 연결하는 쿼리입니다.

---

## 18. 대표 쿼리 2: Q08 LEFT JOIN

INNER JOIN과 LEFT JOIN 차이를 설명합니다.

설명 포인트:

- INNER JOIN은 양쪽에 연결되는 데이터만 보여준다.
- LEFT JOIN은 왼쪽 테이블 데이터를 모두 남긴다.
- Q08에서는 대여 기록이 없는 회원도 결과에 포함된다.
- `COUNT(r.rental_id)`가 0이면 대여 기록이 없는 회원이다.

---

## 19. 대표 쿼리 3: Q15 인덱스

인덱스 선택 이유를 설명합니다.

인덱스:

```sql
CREATE INDEX IF NOT EXISTS idx_rental_member_due
ON rental(member_id, due_date);
```

설명 포인트:

- 특정 회원의 대여 기록을 자주 조회할 수 있다.
- 반납기한 순으로 정렬하는 요구가 있다.
- `member_id`로 먼저 좁히고 `due_date`로 정렬하기 좋다.
- `EXPLAIN QUERY PLAN`으로 인덱스 사용 여부를 확인한다.

## 28. 예상 질문 대응

| 예상 질문 | 핵심 답변 |
|---|---|
| 왜 테이블을 나누었나요? | 중복 제거와 관계 표현을 위해 나누었습니다. |
| PK와 FK는 무엇인가요? | PK는 행 식별자, FK는 다른 테이블 참조값입니다. |
| INNER JOIN과 LEFT JOIN 차이는? | INNER JOIN은 매칭 데이터만, LEFT JOIN은 왼쪽 데이터를 모두 남깁니다. |
| GROUP BY는 무엇인가요? | 같은 기준의 행을 묶어 집계하는 문법입니다. |
| 인덱스는 왜 만들었나요? | 특정 회원의 대여 기록을 반납기한 순으로 빠르게 조회하기 위해 만들었습니다. |
| UPDATE/DELETE는 안전한가요? | PK 기준 WHERE 조건을 사용하고, 자동 실행에서는 임시 DB에서 실행했습니다. |
