# 테이블 설계 노트

이 문서는 B5-1 도서 대여 관리 DB에서 테이블을 왜 `member`, `category`, `book`, `rental` 4개로 나누었는지 설명하기 위한 훈련 문서이다.

평가에서 중요한 것은 테이블 개수 자체가 아니라, **왜 그렇게 나눴는지**, **각 테이블이 어떤 역할을 맡는지**, **FK로 연결한 관계가 실제 도메인에서 무엇을 의미하는지**를 말로 설명하는 것이다.

---

## 1. 한 줄 결론

이 프로젝트는 도서 대여 업무를 아래 네 가지 데이터로 분리했다.

```text
회원 정보     → member
도서 분류     → category
도서 정보     → book
대여 이력     → rental
```

가장 중요한 설계 판단은 `rental`을 별도 테이블로 둔 것이다.

> `rental`은 회원과 도서가 만나는 “대여 사건”을 기록하는 테이블이다.

---

## 2. 왜 한 테이블에 모두 넣지 않았는가?

처음에는 아래처럼 한 표에 모든 정보를 넣을 수도 있다.

| 회원명 | 이메일 | 도서명 | 카테고리 | 저자 | 대여일 | 반납기한 | 상태 |
|---|---|---|---|---|---|---|---|
| 김민준 | minjun.kim@example.com | SQL 첫걸음 | Database | 아사이 아츠시 | 2024-06-01 | 2024-06-15 | RETURNED |
| 김민준 | minjun.kim@example.com | FastAPI 실전 입문 | Backend | 김지훈 | 2024-06-05 | 2024-06-19 | RENTED |
| 이서연 | seoyeon.lee@example.com | 관계형 데이터베이스 설계 | Database | 박성훈 | 2024-06-07 | 2024-06-21 | RETURNED |

이 방식은 엑셀로 보기에는 쉽지만, DB 설계로는 문제가 많다.

### 2.1 중복 저장 문제

같은 회원이 여러 권을 빌리면 회원명과 이메일이 계속 반복된다.

```text
김민준 / minjun.kim@example.com
김민준 / minjun.kim@example.com
김민준 / minjun.kim@example.com
```

같은 도서가 여러 번 대여되면 도서명, 저자, 카테고리도 계속 반복된다.

### 2.2 수정 오류 문제

회원 이메일이 바뀌면 같은 회원이 등장한 모든 행을 수정해야 한다. 한 행이라도 빠뜨리면 같은 회원인데 이메일이 서로 다른 상태가 된다.

### 2.3 삭제 오류 문제

어떤 회원의 마지막 대여 기록을 삭제하면 그 회원 정보까지 같이 사라질 수 있다.

### 2.4 삽입 오류 문제

아직 대여되지 않은 새 책을 등록하고 싶어도, 대여일이나 회원 정보가 없으면 한 표에 넣기 애매하다.

평가 답변:

> 한 테이블에 모두 저장하면 회원 정보와 도서 정보가 반복되고, 수정·삭제·삽입 시 데이터 불일치가 생길 수 있습니다. 그래서 회원, 도서, 카테고리, 대여 이력을 역할별 테이블로 나누었습니다.

## 3. 테이블을 나눈 기준

테이블을 나눌 때는 “독립적으로 관리해야 하는 대상”을 먼저 찾았다.

| 대상 | 독립적으로 관리해야 하는 이유 | 테이블 |
|---|---|---|
| 회원 | 회원은 책을 빌리지 않아도 존재할 수 있다. | `member` |
| 카테고리 | 카테고리는 여러 책을 묶는 기준이다. | `category` |
| 도서 | 도서는 대여 여부와 무관하게 도서관에 존재한다. | `book` |
| 대여 | 대여는 회원이 책을 빌린 사건이다. | `rental` |

핵심 기준:

```text
명사형 대상       → member, category, book
사건/이력형 대상  → rental
```

---

## 4. 각 테이블의 역할

## 4.1 member

`member`는 도서관 회원 정보를 저장한다.

주요 컬럼:

| 컬럼 | 의미 |
|---|---|
| `member_id` | 회원을 식별하는 PK |
| `name` | 회원 이름 |
| `email` | 회원 이메일, UNIQUE |
| `phone` | 연락처 |
| `joined_at` | 가입일 |
| `status` | 회원 상태 |

설계 이유:

> 회원은 도서를 빌리지 않아도 독립적으로 존재할 수 있으므로 별도 테이블로 분리했다.

---

## 4.2 category

`category`는 도서 분류 정보를 저장한다.

주요 컬럼:

| 컬럼 | 의미 |
|---|---|
| `category_id` | 카테고리를 식별하는 PK |
| `name` | 카테고리명, UNIQUE |
| `description` | 카테고리 설명 |

설계 이유:

> 카테고리명을 book 테이블에 문자열로 반복 저장하면 같은 카테고리가 여러 번 중복된다. 그래서 category 테이블로 분리하고, book에서 category_id로 참조하게 했다.

---

## 4.3 book

`book`은 도서 정보를 저장한다.

주요 컬럼:

| 컬럼 | 의미 |
|---|---|
| `book_id` | 도서를 식별하는 PK |
| `category_id` | 도서가 속한 카테고리 FK |
| `title` | 도서 제목 |
| `author` | 저자 |
| `published_year` | 출판년도 |
| `isbn` | ISBN, UNIQUE |
| `price` | 가격 |
| `stock` | 보유 수량 |
| `created_at` | 등록일 |

설계 이유:

> 도서는 대여와 무관하게 도서관에 존재하는 독립 대상이다. 대여 기록과 분리해야 같은 도서의 정보가 여러 번 반복되지 않는다.

---

## 4.4 rental

`rental`은 회원이 도서를 빌린 사건을 저장한다.

주요 컬럼:

| 컬럼 | 의미 |
|---|---|
| `rental_id` | 대여 기록을 식별하는 PK |
| `member_id` | 누가 빌렸는지 나타내는 FK |
| `book_id` | 어떤 책을 빌렸는지 나타내는 FK |
| `rented_at` | 대여일 |
| `due_date` | 반납기한 |
| `returned_at` | 실제 반납일 |
| `status` | 대여 상태 |
| `rental_fee` | 수수료 |

설계 이유:

> `rental`은 회원과 도서를 연결하는 이력 테이블이다. 한 회원은 여러 번 대여할 수 있고, 한 도서도 여러 번 대여될 수 있으므로 대여 사건을 별도 테이블에 기록한다.

---

## 5. 관계 설계

이 DB의 핵심 관계는 3개다.

```text
category 1 : N book
member   1 : N rental
book     1 : N rental
```

---

## 5.1 category 1 : N book

관계:

```text
category.category_id → book.category_id
```

의미:

> 하나의 카테고리에는 여러 권의 책이 속할 수 있다. 그러나 하나의 책은 하나의 카테고리에 속한다.

예시:

```text
Database 카테고리
 ├── SQL 첫걸음
 └── 관계형 데이터베이스 설계
```

평가 답변:

> `category`와 `book`은 1:N 관계입니다. 카테고리 하나가 여러 도서를 가질 수 있기 때문에 `book.category_id`가 `category.category_id`를 참조하도록 설계했습니다.

---

## 5.2 member 1 : N rental

관계:

```text
member.member_id → rental.member_id
```

의미:

> 한 명의 회원은 여러 번 책을 빌릴 수 있다. 그러나 하나의 대여 기록은 반드시 한 명의 회원에게 속한다.

예시:

```text
김민준 회원
 ├── SQL 첫걸음 대여
 ├── FastAPI 실전 입문 대여
 └── 머신러닝 기본기 대여
```

평가 답변:

> `member`와 `rental`은 1:N 관계입니다. 한 회원은 여러 대여 기록을 가질 수 있지만, 하나의 대여 기록은 한 회원에게만 속합니다.

---

## 5.3 book 1 : N rental

관계:

```text
book.book_id → rental.book_id
```

의미:

> 한 권의 책은 시간에 따라 여러 번 대여될 수 있다. 그러나 하나의 대여 기록은 반드시 한 권의 책에 대한 기록이다.

예시:

```text
SQL 첫걸음
 ├── 2024-06-01 김민준 대여
 └── 2024-07-03 임수아 대여
```

평가 답변:

> `book`과 `rental`도 1:N 관계입니다. 한 책은 여러 대여 이력을 가질 수 있으므로, `rental.book_id`가 `book.book_id`를 참조합니다.

---

## 6. rental이 중요한 이유

`rental`은 단순한 연결 테이블이 아니라 “사건 테이블”이다.

`rental`이 없으면 아래 질문에 답하기 어렵다.

```text
누가 어떤 책을 언제 빌렸는가?
아직 반납하지 않은 책은 무엇인가?
연체 상태인 대여 기록은 무엇인가?
회원별 대여 횟수는 몇 번인가?
가장 많이 대여된 책은 무엇인가?
월별 대여 건수는 어떻게 변하는가?
```

`rental`이 있기 때문에 JOIN과 GROUP BY로 실무형 지표를 만들 수 있다.

평가 답변:

> `rental`은 회원과 도서의 관계를 시간과 상태까지 포함해 기록합니다. 단순히 회원과 책을 연결하는 것이 아니라 대여일, 반납기한, 반납일, 상태, 수수료까지 저장하기 때문에 대여 이력 분석이 가능합니다.

## 7. 테이블 분리 전후 비교

### 7.1 분리 전

| 회원명 | 이메일 | 도서명 | 카테고리 | 대여일 | 상태 |
|---|---|---|---|---|---|
| 김민준 | minjun.kim@example.com | SQL 첫걸음 | Database | 2024-06-01 | RETURNED |
| 김민준 | minjun.kim@example.com | FastAPI 실전 입문 | Backend | 2024-06-05 | RENTED |

문제:

```text
회원 정보 반복
도서 정보 반복
카테고리명 반복
회원 이메일 변경 시 여러 행 수정 필요
도서 정보만 따로 관리하기 어려움
```

### 7.2 분리 후

```text
member   : 회원 정보 1번만 저장
category : 카테고리 정보 1번만 저장
book     : 도서 정보 1번만 저장
rental   : 대여 사건만 누적 저장
```

장점:

```text
중복 감소
수정 오류 감소
데이터 무결성 강화
JOIN으로 필요한 결과 조합 가능
GROUP BY로 지표 계산 가능
```

## 8. PK/FK 관점에서 본 설계

### 8.1 PK

PK는 각 행을 유일하게 식별한다.

| 테이블 | PK |
|---|---|
| `member` | `member_id` |
| `category` | `category_id` |
| `book` | `book_id` |
| `rental` | `rental_id` |

평가 답변:

> PK는 테이블 안에서 각 행을 유일하게 구분하는 값입니다. 예를 들어 `member_id`가 있어야 같은 이름의 회원이 있어도 서로 다른 회원으로 구분할 수 있습니다.

---

### 8.2 FK

FK는 다른 테이블의 PK를 참조해 관계를 만든다.

| FK | 참조 대상 | 의미 |
|---|---|---|
| `book.category_id` | `category.category_id` | 이 책이 어느 카테고리에 속하는가 |
| `rental.member_id` | `member.member_id` | 이 대여 기록의 회원은 누구인가 |
| `rental.book_id` | `book.book_id` | 이 대여 기록의 도서는 무엇인가 |

평가 답변:

> FK는 다른 테이블의 PK를 참조해서 관계를 만드는 컬럼입니다. FK 덕분에 존재하지 않는 회원이나 도서를 참조하는 대여 기록을 막을 수 있습니다.

---

## 9. 제약조건을 넣은 이유

| 제약조건 | 적용 예 | 이유 |
|---|---|---|
| `NOT NULL` | `member.name`, `book.title`, `rental.rented_at` | 반드시 있어야 하는 값 |
| `UNIQUE` | `member.email`, `category.name`, `book.isbn` | 중복되면 안 되는 값 |
| `CHECK` | `status`, `price`, `stock`, `rental_fee` | 허용 범위 제한 |
| `FOREIGN KEY` | `book.category_id`, `rental.member_id`, `rental.book_id` | 관계 무결성 보장 |

평가 답변:

> 제약조건은 잘못된 데이터가 들어오는 것을 DB 차원에서 막기 위한 장치입니다. 예를 들어 이메일과 ISBN은 중복되면 안 되므로 UNIQUE를 사용했고, 가격과 재고는 음수가 되면 안 되므로 CHECK를 사용했습니다.

## 10. 이 설계로 풀 수 있는 질문

이 스키마는 다음 요구사항을 SQL로 풀 수 있다.

```text
[ ] 회원별 대여 횟수
[ ] 대여 기록이 없는 회원 찾기
[ ] 카테고리별 도서 수
[ ] 카테고리별 평균 도서 가격
[ ] 연체 기록의 회원명과 도서명 조회
[ ] 최근 대여 기록 조회
[ ] 가장 많이 대여된 도서 TOP 5
[ ] 월별 대여 건수
[ ] 연체율이 높은 회원 찾기
```

이 요구사항들은 `rental`을 중심으로 `member`, `book`, `category`를 JOIN해서 해결한다.

## 11. 평가에서 자주 나오는 질문과 답변

### 질문 1. 왜 테이블을 4개로 나누었나요?

> 회원, 카테고리, 도서, 대여 기록은 각각 독립적으로 관리해야 하는 데이터입니다. 한 테이블에 모두 넣으면 회원 정보와 도서 정보가 반복되고 수정 오류가 생길 수 있습니다. 그래서 `member`, `category`, `book`, `rental`로 나누었습니다.

### 질문 2. 왜 rental 테이블을 따로 만들었나요?

> `rental`은 회원과 도서가 만나는 대여 사건을 기록하는 테이블입니다. 한 회원은 여러 번 대여할 수 있고, 한 도서도 여러 번 대여될 수 있으므로 대여일, 반납기한, 상태 같은 이력을 별도로 저장해야 합니다.

### 질문 3. category를 book 안에 문자열로 넣지 않은 이유는 무엇인가요?

> 카테고리명을 `book` 테이블에 문자열로 반복 저장하면 같은 카테고리명이 여러 번 중복됩니다. 오타나 이름 변경이 생기면 데이터가 불일치할 수 있으므로 `category` 테이블로 분리하고 `book.category_id`로 참조하게 했습니다.

### 질문 4. 회원 정보와 대여 기록을 분리한 이유는 무엇인가요?

> 회원은 아직 책을 빌리지 않아도 존재할 수 있고, 한 회원이 여러 번 대여할 수 있습니다. 회원 정보를 대여 기록마다 반복 저장하지 않기 위해 `member`와 `rental`을 분리했습니다.

### 질문 5. 이 설계에서 가장 중요한 테이블은 무엇인가요?

> 분석 관점에서는 `rental`이 가장 중요합니다. 대여 기록을 중심으로 회원, 도서, 카테고리를 JOIN하면 대여 현황, 연체 현황, 인기 도서, 회원별 대여 횟수 같은 지표를 만들 수 있기 때문입니다.

```text
[ ] member와 rental의 관계는 무엇인가?
[ ] book과 rental의 관계는 무엇인가?
[ ] FK가 없으면 어떤 문제가 생기는가?
[ ] UNIQUE 제약조건은 왜 필요한가?
[ ] CHECK 제약조건은 왜 필요한가?
```

## 13. 평가 직전 핵심 문장

아래 문장을 자연스럽게 말할 수 있으면 충분하다.

> 이 프로젝트는 도서 대여 도메인을 `member`, `category`, `book`, `rental` 네 테이블로 분리했습니다. 회원과 도서 정보는 반복 저장하지 않고 각각 독립 테이블에 저장했습니다. `rental`은 누가 어떤 책을 언제 빌렸는지를 기록하는 대여 이력 테이블입니다. `book.category_id`, `rental.member_id`, `rental.book_id`를 FK로 연결해 존재하지 않는 카테고리, 회원, 도서를 참조하지 못하게 했습니다. 이렇게 분리했기 때문에 JOIN으로 사람이 읽기 쉬운 결과를 만들고, GROUP BY로 회원별 대여 횟수나 카테고리별 도서 수 같은 지표를 계산할 수 있습니다.
