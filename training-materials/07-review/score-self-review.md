# B5-1 자가 평가 및 점수 리뷰

이 문서는 B5-1 「SQL로 만드는 나만의 데이터베이스」 프로젝트를 제출하기 전 스스로 점수를 매기고, 감점 위험을 찾아 보완하기 위한 자가 평가 문서이다.

핵심 목적은 다음 세 가지다.

```text
1. 필수 요구사항을 빠짐없이 충족했는지 확인한다.
2. 평가자가 볼 근거 파일을 명확히 연결한다.
3. 감점 가능성이 있는 부분을 제출 전 보완한다.
```

---

## 1. 현재 자가 평가 결론

현재 프로젝트가 아래 조건을 모두 만족한다면 **90점 이상**, 실행 검증과 말하기 답변까지 안정적이면 **95점 이상**을 목표로 볼 수 있다.

```text
[ ] 4개 이상 테이블이 있다.
[ ] 1:N 관계 2개 이상이 있다.
[ ] PK/FK/NOT NULL/UNIQUE/CHECK 제약조건이 있다.
[ ] 각 테이블에 충분한 샘플 데이터가 있다.
[ ] SELECT, JOIN, GROUP BY, 서브쿼리, UPDATE, DELETE, 인덱스 쿼리가 있다.
[ ] 실행 결과가 남아 있다.
[ ] ERD가 있다.
[ ] 평가 답변을 말로 설명할 수 있다.
```

자가 평가 문장:

> 이 프로젝트는 B5-1 필수 요구사항을 대부분 충족합니다. 특히 4개 테이블과 3개의 1:N 관계, PK/FK/제약조건, 핵심 SQL 쿼리, 실행 결과, ERD, 평가 답변을 모두 갖추고 있어 실행 검증이 통과한다면 90점 이상을 기대할 수 있습니다.

## 5. 감점 위험 진단

| 위험 | 감점 가능성 | 보완 방법 |
|---|---:|---|
| SQL은 있는데 왜 그렇게 설계했는지 설명 못함 | 높음 | `expected-questions.md`, `answer-scripts.md` 반복 |
| JOIN 조건을 FK 기준으로 말하지 못함 | 높음 | JOIN 조건 3개 암기 |
| GROUP BY와 HAVING 차이 혼동 | 중간 | WHERE/HAVING 답변 연습 |
| UPDATE/DELETE 안전성 언급 누락 | 중간 | SELECT 선확인, WHERE 조건 강조 |
| 인덱스를 “빠르니까 만들었다”고 설명 | 중간 | 조회 시나리오와 연결 |
| 실행 결과 누락 | 높음 | `run_all.sh` 재실행 후 결과 갱신 |
| README 실행 방법 불명확 | 중간 | 실행 명령 확인 |
| ERD와 실제 schema 불일치 | 높음 | FK 기준으로 다시 비교 |

## 6. 점수 향상 우선순위

### 1순위: 실행 검증

```bash
chmod +x scripts/run_all.sh
./scripts/run_all.sh
```

확인:

```text
[ ] 오류 없이 끝난다.
[ ] 결과 파일이 최신이다.
[ ] UPDATE/DELETE가 임시 DB에서 안전하게 검증된다.
```

점수 영향:

> 실행 검증은 가장 큰 감점 방지 요소다. 쿼리가 실제로 돌아간다는 증거가 있어야 한다.

---

반드시 말할 것:

```text
[ ] 왜 4개 테이블인가?
[ ] rental은 왜 필요한가?
[ ] 1:N 관계는 무엇인가?
[ ] INNER JOIN과 LEFT JOIN 차이는 무엇인가?
[ ] GROUP BY는 무엇인가?
[ ] 서브쿼리는 무엇인가?
[ ] 인덱스는 왜 rental(member_id, due_date)인가?
```

점수 영향:

> 구현이 되어 있어도 설명하지 못하면 점수가 낮아질 수 있다.

---

### 3순위: 문서와 SQL 일치 확인

```text
[ ] ERD의 FK와 schema.sql의 FK가 일치한다.
[ ] evaluation_answers.md의 설명과 실제 쿼리가 일치한다.
[ ] README의 실행 방법이 실제로 동작한다.
[ ] results/query_results.md가 현재 쿼리와 맞다.
```

점수 영향:

> 문서와 코드가 다르면 신뢰도가 떨어진다.

## 7. 평가자 관점 리뷰

평가자는 보통 다음을 빠르게 확인한다.

```text
1. 저장소가 정리되어 있는가?
2. README를 보고 실행할 수 있는가?
3. 스키마가 요구사항을 만족하는가?
4. 샘플 데이터가 충분한가?
5. 핵심 쿼리가 빠짐없이 있는가?
6. 실행 결과가 있는가?
7. ERD가 실제 스키마와 맞는가?
8. 본인이 설계 이유를 설명할 수 있는가?
```

따라서 제출 전 스스로 이렇게 말할 수 있어야 한다.

> 평가자가 README에서 실행 방법을 보고, SQL 폴더에서 스키마·데이터·쿼리를 확인하고, results 폴더에서 실행 결과를 확인할 수 있도록 구성했습니다. ERD와 평가 답변도 함께 제공해 설계 의도를 설명할 수 있게 했습니다.

## 10. 평가 직전 10문장 점검

아래 10문장을 막힘없이 말할 수 있어야 한다.

```text
1. 이 프로젝트는 SQLite 기반 도서 대여 관리 데이터베이스입니다.
2. 테이블은 member, category, book, rental 네 개입니다.
3. category-book, member-rental, book-rental은 1:N 관계입니다.
4. rental은 회원과 도서의 대여 이력을 기록하는 테이블입니다.
5. PK는 각 행을 식별하고 FK는 테이블 관계를 만듭니다.
6. INNER JOIN은 매칭되는 행만, LEFT JOIN은 왼쪽 행을 유지합니다.
7. GROUP BY는 같은 기준의 행을 묶고 COUNT/SUM/AVG로 집계합니다.
8. 서브쿼리는 안쪽 SELECT 결과를 바깥 쿼리 조건으로 사용합니다.
9. UPDATE/DELETE는 SELECT로 대상 확인 후 WHERE 조건을 명확히 써야 합니다.
10. idx_rental_member_due는 회원별 반납기한 조회를 위한 복합 인덱스입니다.
```
